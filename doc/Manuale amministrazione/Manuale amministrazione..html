  1. Introduzione

Il sistema CRL Gestione Atti si compone di una web application J2EE front-end, installata su un Application Server Apache Tomcat 7.0.34 ed un istanza Alfresco 4.2.c Community Edition con DataBase MySQL 5.5.28 installata su un Application Server Apache Tomcat 7.0.37.

  1. Configurazione Alfresco

Per configurare Alfresco occorre partire da una nuova istanza di Alfresco community 4.2.c e di Apache Tomcat 7.0.37 ed eseguire i seguenti passi:

  • Estrarre gli archivi zip di Apache Tomcat 7.0.37 ed Alfresco Community 4.2.c.
  • Installare Apache Tomcat seguendo la procedura standard.
  • Creare la cartella shared/classes all’interno della cartella tomcat.
  • Modificare il file <TOMCAT-HOME>/conf/catalina.properties ed impostare il valore della proprietà shared.loader come riportato di seguito: shared.loader=${catalina.base}/shared/classes
  • Copiare il driver JDBC per il database MYSQL nella cartella lib.
  • Modificare il file <TOMCAT_HOME>/conf/server.xml, impostare la porta http al valore 8082 ed abilitare l’enconding UTF-8 aggiungendo la stringa URIEncoding="UTF-8 al connettore:

<Connector port="8082" protocol="HTTP/1.1" URIEncoding="UTF-8"

               connectionTimeout="20000"

               redirectPort="8445" />

  1. Integrazione Atti Indirizzo

Il sistema di Gestione Atti di Indirizzo e Sindacato Ispettivo è un’altra istanza Alfresco presente in CRL e gestita dalla Fondazione Politecnico. Il sistema gestisce alcune tipologie di atti (MOZ, RIS ecc..) che non sono gestiti all’interno della Nuova Gestione Atti.

L’integrazione sviluppata permette di “collegare” gli atti presenti nel Nuovo Gestione Atti con gli atti presenti in Gestione Atti di Indirizzo e Sindacato Ispettivo.

L’integrazione è realizzata mediante lo scambio di file xml che modellano le informazioni relative agli atti che i due sistemi si devono scambiare. Per i dettagli sui file xml consultare il documento “Integrazione NGA-GASI”.

L’integrazione è stata sviluppata con le seguenti componenti:

1. Regola sullo space che contiente gli atti “ContenitoreAtto” che in fase di creazione dell’atto prende il nome del tipo atto (PDL, PDA ecc...).

Nello spazio c’è una regola che si innesca sull’update del contenuto di tipo crlatti:atto (e sottotipi) ed esegue lo script “creaXMLExportToAttiIndirizzo.js” che genera il file xml da trasmettere e lo salva nella cartella Company Home/Export/Gestione Atti/AttiIndirizzo.

2. Invio dei file xml. Lo script “attiIndirizzoExporter.js” prende i file presenti nella cartella Company Home/Export/Gestione Atti/AttiIndirizzo, li invia mediante CMIS all’istanza Alfresco Atti Indirizzo. Le configurazioni dei parametri per l’invio sono nel file config/context/atti-indirizzo-export-job-script-context.xml. Nello stesso file di configurazione ci sono i parametri per la schedulazione del processo di trasmissione.

3. Import dei file xml. L’istanza di Alfresco Atti Indirizzo genera dei file xml che trasmette mediante CMIS nella cartella  Company Home/Import/Gestione Atti/AttiIndirizzo. Lo script “attiIndirizzoImporter.js” prende tutti i file presenti nella cartella, effettua il parsing e genera i contenuti di tipo crlatti:attoIndirizzo. Successivamente vengono create delle associazioni con i contenuti di tipo crlatti:atto mediante interfaccia (collegamenti). La  configurazione e la schedulazione del processo di parsing è nel file config/context/atti-indirizzo-import-job-script-context.xml.

  1. Integrazione Protocollo

Il Nuovo Gestione Atti espone dei servizi REST (webscript) che permettono al sistema di protocollo (Archiflow) la creazione dei nuovi atti e la trasmissione di allegati relativi ad atti già creati.

La logica di funzionamento interna ad Alfresco è la seguente:

a) Protocollo chiama mediante POST il servizio “/crl/protocollo/atto”. Il servizio memorizza l’atto creato nella cartella company home/import/Gestione Atti/Protocollo/Atti.

b) Una regola in ascolto sulla cartella di importazione, sposta l’atto appena creato all’interno della corretta cartella del repository. Lo stesso meccanismo è implementato per la gestione degli allegati.

  1. Integrazione Anagrafica

L’integrazione con il sistema di gestione anagrafica avviene mediante query su database SQLServer 2008 situato all’interno della intranet di CRL.

Lo script che innesca l’infasamento (anagraficaImport.js) è schedulato una volta al giorno.

La configurazione si trova nel file config/context/anagrafica-import-job-script-context.xml.

Lo space Anagrafica si trova all’interno di “company home/CRL” e presente una serie di sottocartelle in cui sono presenti:

- Consiglieri (cancellati e ricreati ogni volta che viene eseguito lo script di infasamento)

- ConsiglieriStorici (non vengono mai cancellati ma solo aggiunti in maniera incrementale)

- Commissioni: insieme ai nodi dell’anagrafica sono creati anche i gruppi utente relativi alle diverse commissioni con la convenzione NN_NomeCommissione, dove NN è il numero romano che identifica la legislatura (es: “IX_Commissione IV Affari Interni”).

- Gruppi Politici

- Legislature

Primo infasamento

Dopo aver fatto girare lo script la prima volta, verranno create solo le legislature all’interno della cartella Legislature in Anagrafica.

Occorre entrare nelle proprietà della legislatura corrente ed editare il flag “Corrente”. In questo modo quando girerà nuovamente lo script di infasamento, verranno creati tutti i nodi relativi ai consiglieri e alle commissioni con riferimento alla legislatura attivata.

Chiusura Legislatura

Quando occorre chiudere una legislatura sarà necessario editare il flag “Chiudi Legislatura”. In questo modo lo script “chiudiLegislatura.js” prenderà tutti gli atti della legislatura e cambierà lo stato in chiuso con l’opportuna motivazione. Poi occorrerà disabilitare il flag “Corrente” e abilitarlo sulla nuova legislatura.

  1. Gestione Gruppi e Utenze

In fase di installazione sono creati, in automatico durante il bootstrap, ii seguenti gruppi:

- GROUP_ServizioCommissioni

- GROUP_Aula

- GROUP_Commissioni

- GROUP_CPCV

- GROUP_CRLAdmin

- GROUP_CRLGuest

Dopo aver installato il modulo occorre far girare manualmente lo script di infasamento anagrafica e verranno create in automatico i gruppi relativi alle commissioni presenti in anagrafica.

I gruppi commissione (es: “GROUP_IX_Commissione I Affari e Bilancio”) sono creati come sottogruppi del gruppo “Commissioni” già creato in fase di bootstrap.

Dopo aver creato i gruppi, è necessario creare le utenze. Per la fase di sperimentazione e test sono state create le utenze corrispondenti ai relativi gruppi, come riportato di seguito:

- GROUP_ServizioCommissioni [user: servizioCommissioni, psw: servizioCommissioni]

- GROUP_Aula [user: aula, psw: aula]

- GROUP_Commissioni

- GROUP_IX_Commissione I Affari e Bilancio [user: commissione1, psw:commissione1]

- GROUP_IX_Commissione II [user: commissione2, psw: commissione2]

- GROUP_Giunta Regolamento [user:giuntaRegolamento, psw: giuntaRegolamento]

- GROUP_CPCV [user:cpcv, psw: cpcv]

- GROUP_CRLAdmin  [user:crladmin, psw: crladmin]

- GROUP_CRLGuest  [user:crladmin, psw: crladmin]


N.B. La giunta regolamento si comporta come una commissione a tutti gli effetti.
