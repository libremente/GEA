 Executive Summary

Questo documento descrive le convenzioni ed i metodi di interfacciamento previsti dal repository degli atti del Consiglio della Regione Lombardia per il sistema di protocollazione.

RESTful API del dominio CRL per protocollo

Questo capitolo descrive tutti i metodi esposti dal repository per poter fare in modo di ricevere o aggiornare gli atti dal sistema di protocollazione afferente al dominio di interesse del CRL.

Convenzioni

In questo paragrafo sono descritte alcune convenzioni utilizzate in questo documento per la rappresentazione del dominio, i parametri scambiati tra i sistemi applicativi e gli standards adottati.

Identificazione dei contenuti

Sarà preso come identificativo per l’interscambio delle informazioni l’id interno del sistema di protocollazione che verrà mantenuto all’interno del repository e verrà associato sia agli atti, che ai relativi allegati. In questo modo il sistema di protocollo potrà agire senza conoscere alcun identificativo specifico del repository.

Standards

L’architettura di riferimento per questa interfaccia sarà basata sulla specifica REpresentational State Transfer (REST) utilizzando direttamente i metodi del protocollo HTTP.

Il formato che sarà utilizzato per l’invio dei metadati dell’atto è JavaScript Object Notation (JSON).

Classificazione dei metodi di backend per protocollo

La REST API di dominio CRL - protocollo sarà categorizzata all’interno del repository Alfresco secondo i seguenti criteri di classificazione Spring WebScripts:

family: CRL - Gestione Atti - Protocollo

package: crl/gestioneatti/protocollo

Indirizzamento

Il formato della URL da chiamare per ogni metodo descritto in questo capitolo avrà come context root il seguente:

http://<IP>:<PORTA>/alfresco/service/

Quindi ad esempio per poter invocare una richiesta di autenticazione sarà necessario invocare una richiesta HTTP POST con la seguente URL:

http://<IP>:<PORTA>/alfresco/service/api/login

Specifica di invocazione dei servizi

In questo documento, per ogni metodo esposto si farà sempre riferimento al protocollo HTTP ed all’invocazione dei metodi GET, POST, DELETE e PUT ed alla relativa URI (URL) di riferimento.

Ad esempio se il metodo è esposto tramite un servizio HTTP che risponde a richieste di metodi GET e ha come URL crl/protocollo/atto, questa definizione verrà descritta nel seguente modo:

GET crl/protocollo/atto

In seguito verranno descritti i parametri necessari per poter effettuare una richiesta al metodo ed il formato richiesto. Sarà inoltre fornita una descrizione del formato in output per ogni servizio e dei relativi codici di risposta HTTP supportati.

Servizi esposti

In questo paragrafo sono descritti tutti i servizi esposti dal repository per poter far interagire in modo corretto il sistema esterno di protocollazione.

Login

Sarà creata un’apposita utenza protocollo per poter effettuare la prima richiesta di autenticazione al repository. Questa richiesta dovrà essere eseguita prima di ogni altra interazione verso Alfresco. La durata di sessione utente è di 60 minuti.

Definizione del metodo:

POST api/login

Parametri in JSON:

  • username
  • password

Esempio di formattazione JSON per una richiesta di autenticazione dell’utente protocollo:

{
    "username" : "protocollo",
    "password" : "mypassword"
  }

Risposta:

  • 200 - OK: restituisce inoltre il ticket di sessione utente
  • 400 - bad request: per mancanza di uno dei parametri
  • 403 - forbidden: per fallita autenticazione

Esempio di risposta in JSON:

{

  "data":

  {

   "ticket":"TICKET_4a5f13f5111c925c37b7d5fec42debb4cc16812b"

  }

}

Il ticket di sessione restituito dal repository dovrà essere utilizzato per ogni successiva richiesta per fare in modo che Alfresco utilizzi la stessa sessione utente con i relativi settaggi di autorizzazione per i contenuti.

Per poter effettuare le successive richieste con la stessa sessione utente sarà necessario concatenare alla URL di chiamata il valore del ticket attraverso l’utilizzo del parametro alf_ticket in query string, nel seguente modo:

crl/protocollo/atto?alf_ticket=TICKET_4a5f13f5111c925c37b7d5fec42debb4cc16812b

Logout

Il metodo di logout permette di rimuovere una sessione utente attiva all’interno del repository.

In seguito all’invocazione di questo metodo il ticket di sessione risulterà invalido e l’utente non potrà più interagire con il repository.

Definizione del metodo:

DELETE api/login/ticket/<VALORE DEL TICKET>?alf_ticket=<VALORE DEL TICKET>

Parametri richiesti nella URL ed in query string:

  • valore del ticket di sessione utente attivo

Risposta:

  • 200 - OK: la sessione utente è stata rimossa dal repository
  • 400 - bad request: per mancanza di uno dei parametri
  • 403 - forbidden: per fallita autenticazione

Creazione / aggiornamento di un atto

Questo metodo permette la creazione e l’aggiornamento di un atto in merito alla gestione dei relativi metadati. L’esecuzione di questo metodo andrà a manipolare il nodo afferente ad un atto all’interno del repository.

Se questo stesso metodo verrà invocato per modificare alcuni metadati di un atto esistente, i parametri di riferimento per poter identificare lo stesso atto all’interno del repository saranno i seguenti:

  • idProtocollo
  • numeroAtto

In questo modo verrà eseguita una query all’interno del repository per verificare se l’atto è già presente. Nel caso in cui l’atto non venisse trovato, il sistema automaticamente genererà un nuovo nodo per l’atto.

Definizione del metodo:

POST crl/protocollo/atto?alf_ticket=<VALORE DEL TICKET>

Parametri in JSON:

  • numeroAtto
  • estensioneAtto
  • tipologia
  • tipoAtto
  • legislatura
  • idProtocollo: id interno del sistema di protocollo
  • numeroProtocollo
  • numeroRepertorio
  • dataRepertorio
  • urlFascicolo
  • classificazione
  • oggetto
  • tipoIniziativa
  • dataIniziativa
  • descrizioneIniziativa
  • numeroDgr
  • dataDgr
  • assegnazione
  • esibenteMittente: valorizzato come descritto all’interno del documento #1

Esempio di formattazione JSON per una richiesta:

{

            "atto" : {

                    "numeroAtto" : "148",

                 "estensioneAtto" : "bis",

                    "tipoAtto" : "PDL",

                    "tipologia" : "Generale Entrata",

                    "legislatura" : "XI",

                    "idProtocollo" : "<ID INTERNO DEL SISTEMA PROTOCOLLO>",

                    "numeroProtocollo" : "0003522/12",

                    "numeroRepertorio" : "",

                    "dataRepertorio" : "2012-02-29",

                 "urlFascicolo" : "/url/relativa/al/fascicolo",

                    "classificazione" : "",

                    "oggetto" : "NORME PER ELEZIONE PRESIDENTE...",

                    "tipoIniziativa" : "01_ATTO DI INIZIATIVA CONSILIARE",

                    "dataIniziativa" : "2012-08-09",

                    "descrizioneIniziativa" : "",

                    "numeroDgr" : "",

                    "dataDgr" : "2012-09-10",

                    "assegnazione" : "A6 _ SERVIZIO COMMISSIONI",

                    "esibenteMittente" : "CONSIGLIERI REGIONALI: ANGELO GALLI, CARLO CECCHETTI (PDL) - VALERIO BIANCHI (IDV) - MARIO CIOCCA (PD)"

                    }

            }

I campi evidenziati in rosso dovranno sempre essere valorizzati. Gli altri campi dovranno essere sempre presenti nel JSON ma potranno avere un valore blank.

Risposta:

  • 200 - OK
  • 400 - bad request: per mancanza di uno dei parametri obbligatori:
  • numeroAtto
  • tipoAtto
  • legislatura
  • idProtocollo
  • 403 - forbidden: per fallita autenticazione
  • 500 - Internal error

Supporto del tipo di atto EAC

Il tipo di atto EAC non dipende da alcuna legislatura.

Esempio di formattazione JSON per una richiesta relativa ad un atto di tipo EAC:

{

"atto" : {

                    "numeroAtto" : "148",

                    "tipoAtto" : "EAC",

                    "idProtocollo" : "<ID INTERNO DEL SISTEMA DI PROTOCOLLO>",

                    "dataAtto" : "2012-10-15",

                    "note" : "Prova nota di un atto di tipo EAC"

                    }

    }

I campi evidenziati in rosso dovranno sempre essere valorizzati. Gli altri campi dovranno essere sempre presenti nel JSON ma potranno avere un valore blank.

I campi mostrati sopra sono quelli espressamente richiesti dalla tipologia di atto EAC, eventuali altri campi valorizzati dell’atto, potranno essere presenti nel JSON, ma non saranno presi in considerazione dalla logica di backend.

Inserimento di un allegato

Questo servizio permette di eseguire l’upload per l’inserimento di un allegato afferente ad uno specifico atto.

La richiesta HTTP dovrà essere eseguita considerando le seguenti caratteristiche:

  • enctype:"multipart/form-data"
  • accept-charset:"utf-8"

Parametri in richiesta HTTP ():

  • idProtocolloAtto: id protocollo relativo all’atto
  • idProtocolloAllegato: id protocollo relativo all’allegato
  • file: contenuto binario dell’allegato (parametro File nella richiesta HTTP)

Esempio di form HTML compatbile con la richiesta:

<form action="/alfresco/service/crl/protocollo/atto/allegato/insert?alf_ticket=<VALORE DEL TICKET>" method="post" enctype="multipart/form-data" accept-charset="utf-8">

    <table>

            <tr>

                    <td>ID protocollo atto:</td>

                    <td>

                            <input type="text" name="idProtocolloAtto">

                    </td>

            </tr>

            <tr>

                    <td>ID protocollo allegato:</td>

                    <td>

                            <input type="text" name="idProtocolloAllegato">

                    </td>

            </tr>

            <tr>

                    <td>File:</td>

                    <td>

                            <input type="file" name="file">

                    </td>

            </tr>

            <tr>

                    <td></td>

                    <td><input type="submit" value="Salva allegato"></td>

            </tr>

    </table>

    </form>

Risposta:

  • 200 - OK
  • 400 - bad request: per mancanza di uno dei parametri obbligatori:
  • idProtocolloAtto
  • idProtocolloAllegato
  • file
  • 403 - forbidden: per fallita autenticazione
  • 500 - Internal error
